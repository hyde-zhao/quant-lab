## 2. 推荐 HLD

推荐方案：项目自有多因子研究闭环。

核心结论：

- 外部项目只作为 `reference_only` / `optional_spike` / `exclude` / `forbidden_migration`，不得成为默认 framework、truth、provider、runner、optimizer 或 report truth。
- schema 和校验不从零设计：以 `research_input_v1`、实验 17-21 `FactorDefinition`、CR-011 factor panel audit、label window gate、Stage6 admission gate 为基线，用 Qlib / Alphalens / Zipline / LEAN cross-check，并 fail-closed。
- HLD 定义 `FactorSpec`、`FactorRunSpec`、`FactorPanelContract`、`LabelWindowSpec`、`FactorEvaluationReport`、`MultiFactorCombiner`、`MultiFactorPortfolioPlan`、`ExperimentManifest`、`ResearchReportCatalog`、`StrategyAdmissionPackage` 和 `order_intent_draft_v1` handoff。
- `StrategyAdmissionPackage` 只输出准入证据、blocked reasons 和 draft handoff；真实 QMT / simulation / live 仍由 CR-020..CR-024 独立授权。

## 3. 备选方案

| 方案 | 结论 | 使用条件 |
|---|---|---|
| CR30-B：Qlib runner-first / qrun 集成 | 不推荐作为 CR-030 默认方案；保留为 CR-026 后续 Spike | CR-030 合同冻结后，用户单独批准依赖隔离、provider 禁用、运行授权和回归范围。 |
| CR30-C：文档 / Spike-only | 不推荐主线；可作为 CP3 不通过时回退 | 用户暂不接受合同冻结，或许可证 / 依赖风险需要先独立评估。 |
| CR30-D：外部框架适配层 / adapter-first | 条件推荐为后续增强，不进入 P0 | 内部合同稳定后出现明确外部 runner / analyzer 导入需求。 |

## 6. 关键取舍

| 维度 | 推荐取舍 |
|---|---|
| 复杂度 | 接受中等合同设计复杂度，换取 truth、gate、report 和 admission 的一致性。 |
| 成本 | 当前只写 HLD 和 CP3 预检，不新增实现 / 依赖成本。 |
| 扩展性 | 通过后续 CR-026 / optimizer Spike / vectorbt / PyBroker 等分流保留扩展。 |
| 可验证性 | 使用字段字典、错误码、fail-closed 和 TS-030-01..10 支撑后续验证。 |
| 维护成本 | 避免外部框架接管主路径，减少依赖和迁移债。 |
| 平台兼容 | 不引入 Qlib runtime、Notebook、provider 或外部 runner，保持本项目现有平台边界。 |
| 安全风险 | CP3 通过不授权实现、外部运行、provider/lake/publish、QMT/simulation/live 或凭据。 |

## 7. 风险

| 风险 | 等级 | 缓解 |
|---|---|---|
| 外部框架被误设为默认 truth / runner | 高 | CP3 DQ 明确 CR30-A；外部项目仅 reference / Spike。 |
| schema 字段遗漏 | 高 | 以既有基线 + Qlib / Alphalens / Zipline / LEAN cross-check；CP3 review 可要求补字段。 |

| 准入包误读为交易授权 | 高 | `StrategyAdmissionPackage` 输出 blocked reasons 和 draft handoff；QMT 另 CR。 |
| 静态调研遗漏 runtime 细节 | 中 | runtime 细节仅作为后续 Spike，不作为 CP3 阻断。 |
