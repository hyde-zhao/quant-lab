# 8 维度覆盖框架（Phase 2）

> 仅在 `use-case-discovery` 的 Phase 2 加载。
> 目标不是“重新发现全部场景”，而是对已有基线做**遗漏扫描**。
> 进入本阶段前，应已完成 Phase 1A / 1B，并在 `USE-CASES.md` 中写入 `target_artifact_type`、`governance_mode`、`review_policy`。

## 使用规则

1. 先基于 Phase 1 草稿识别遗漏，再决定是否追问；不要脱离基线空泛发散。
2. 每轮最多追问 3 个维度，避免用户疲劳。
3. 每个维度必须写入覆盖状态：`已覆盖 / 已补充 / 不适用 / 待调研`。
4. 用户可临时追加最多 2 个自定义维度；超过 2 个时提示走 CR 将其正式化。
5. 若连续多轮无新增，可建议进入确认阶段，但不能把未处理维度静默省略。
6. 自定义维度 `Dx` 只用于会话级补充；若后续需要长期保留，应在确认后通过变更流程正式化。

## 默认 8 维

| 维度 ID | 维度名称 | 核心检查点 | 常见遗漏 | 追问示例 |
|---|---|---|---|---|
| D1 | 用户维度 | 是否还有不同角色、权限、经验水平的使用者 | 管理员、只读用户、外部协作者、新手 / 高阶用户 | “除了主要使用者，还有谁会查看、审批、配置或维护这件事？” |
| D2 | 任务维度 | 同一目标在不同任务阶段是否有不同场景 | 初始化、批量处理、迁移、回滚、审计 | “在首次建立、批量操作或迁移旧数据时，流程会变吗？” |
| D3 | 动机维度 | 用户为什么现在要做这件事、最在意什么结果 | 节省时间、降低错误率、减少学习成本、提高信心 | “如果这个场景处理得好，最希望改善的指标是什么？” |
| D4 | 时间维度 | 首次、日常、偶发、升级、退出是否存在差异 | 首次配置、低频但高风险操作、版本升级、停用迁移 | “第一次使用和日常重复使用，有哪些明显不同？” |
| D5 | 环境维度 | 使用环境、设备、网络与运行上下文是否变化 | CI/CD、离线、受限权限、多平台、跨时区 | “这个场景会只在本地发生，还是也会在 CI、远程机或弱网环境发生？” |
| D6 | 方式维度 | 输入方式、操作入口、执行路径是否不止一种 | 命令行、配置文件、交互式向导、API、批处理 | “用户会手动操作、传配置文件，还是让别的系统自动触发？” |
| D7 | 异常维度 | 输入异常、权限不足、依赖失败时如何处理 | 缺失参数、格式错误、超时、外部服务不可达、权限不足 | “如果输入缺失、格式错了或外部依赖失败，用户预期系统怎么反馈？” |
| D8 | 集成维度 | 还有哪些外部系统、流程或上下游要协作 | Git、CI、审批流、其他 Skill / Agent、第三方服务 | “这个场景需要把结果交给谁，或依赖哪些外部系统才能完成？” |

## 旅程覆盖矩阵

| 旅程节点 | 重点维度 | 提醒 |
|---|---|---|
| 首次接触 / 首次配置 | D1、D3、D4、D5、D6 | 不要只问理想路径，要覆盖 onboarding 成本与首次失败反馈 |
| 日常高频使用 | D2、D3、D5、D6 | 关注效率、批量化、默认输入与重复操作 |
| 偶发但高风险操作 | D2、D4、D7、D8 | 常见于迁移、回滚、权限变更、批量变更 |
| 异常恢复 / 故障处理 | D5、D7、D8 | 重点问错误提示、补救路径、人工介入点 |
| 升级 / 停用 / 退出 | D4、D7、D8 | 容易遗漏版本兼容、数据迁移与退出交接 |

## 自定义维度追加示例

| 自定义维度 | 适用信号 | 示例追问 |
|---|---|---|
| 合规 / 审计 | 医疗、金融、政企、强审计要求 | “这个场景是否需要留痕、审批或满足合规校验？” |
| 性能 / 时延 | 批量任务、实时反馈、SLA 明确 | “这个场景对时延、吞吐或超时阈值有硬要求吗？” |
| 国际化 / 本地化 | 多语言、多地区、多时区 | “同一场景在不同语言、地区或时区下会有差异吗？” |

## 追问收敛规则

- 若某维度已被现有场景完整覆盖，只记录“已覆盖”，不重复追问。
- 若用户明确表示不涉及该维度，记录为“不适用”，并补一句理由。
- 若用户知道有缺口但当前无法回答，记录为“待调研”，避免硬凑答案。
- 若新增的是**新画像或新场景**，优先回退到 Phase 1；若只是补某维遗漏，留在 Phase 2 定向补齐。
